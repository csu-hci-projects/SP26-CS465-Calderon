# AirDesk Pilot Plan

## Purpose

Run a small formative pilot with Caden and one roommate. This is enough for a solo-project system paper, but the paper must frame it as pilot evidence rather than broad validation.

## Participants

- Participant 1: Caden, project developer.
- Participant 2: roommate, non-developer pilot participant.

## Conditions

1. Keyboard/mouse baseline at desk.
2. AirDesk live dry-run.
3. Optional AirDesk guarded execute mode, only if dry-run is stable.
4. Optional simulated situational impairment:
   - standing away from keyboard,
   - wearing gloves,
   - holding an object,
   - or another safe "do not touch keyboard" setup.

## Candidate Tasks

Use 3-5 tasks only.

- Switch workspace left.
- Switch workspace right.
- Move focus left.
- Move focus right.
- Cancel or recover from listening mode.
- Optional: pause/resume media or runtime.

Avoid text entry and small-target cursor tasks in the first paper unless cursor mode becomes the paper focus.

## Measures

Objective:

- task success,
- task completion time,
- false activations,
- missed intended gestures,
- repeated fires,
- gesture-to-action latency,
- fallback keyboard/mouse use,
- tracking loss or setup failure.

Subjective:

- perceived control,
- fatigue/discomfort,
- frustration,
- perceived usefulness,
- preference versus baseline,
- short interview notes.

## Minimal Trial Script

For each participant:

1. Explain that AirDesk is a prototype and can be stopped any time.
2. Run a short practice phase.
3. Run keyboard/mouse baseline tasks.
4. Rest.
5. Run AirDesk dry-run tasks.
6. Optionally run simulated situational impairment.
7. Ask short interview questions.

## Short Interview Questions

- Which actions felt natural?
- Which actions felt unreliable or confusing?
- Did the clutch/listening mode make the system feel safer?
- Did the interaction feel tiring?
- When would you actually prefer this over keyboard/mouse?
- What would need to change before this felt usable?

## Abort Rules

- Stop if false activations occur repeatedly during normal motion.
- Stop if the participant reports fatigue or discomfort.
- Stop real execution if the wrong workspace or focus target is controlled twice.
- Stop if pause does not suppress actions.
